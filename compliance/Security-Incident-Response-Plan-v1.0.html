1. Purpose and Scope

This Security Incident Response Plan defines the procedures Marketmate Solutions Ltd follows when a security incident is detected affecting the e-Lister platform, its customers, or any personal data processed through the platform.

This plan applies to all security incidents including:

  • Unauthorised access to customer data or marketplace credentials
  • Personal data breaches affecting buyer PII retrieved via marketplace APIs
  • Compromise of API credentials, OAuth tokens, or encryption keys
  • Ransomware, malware, or other destructive attacks
  • Denial of service attacks affecting platform availability
  • Insider threats or accidental data exposure

4. Response Procedures

Phase 1 – Detection and Reporting

Security incidents may be detected via:

  • GCP Cloud Audit Log alerts (Data Read/Write anomalies on Firestore or Secret Manager)
  • Trivy or Semgrep scan findings from automated CI/CD pipeline
  • Marketplace notification (Amazon, eBay, Shopify security alert)
  • Manual discovery during development or operations
  • External report from a customer, researcher, or third party

Any suspected incident must be reported immediately to Steven Adams via mrstevenradams@gmail.com.

Phase 2 – Containment

  • Revoke any compromised credentials via GCP Secret Manager and respective developer portals
  • Disable affected Cloud Run service revisions if the service is compromised
  • Apply GCP firewall rules to block suspicious traffic sources
  • Preserve all relevant GCP Cloud Audit Logs by exporting to a secure GCS bucket before any remediation
  • Notify affected marketplaces via their developer portals

Phase 3 – Investigation

  • Review GCP Cloud Audit Logs to identify the scope of access
  • Determine which data categories were affected and the number of data subjects involved
  • Identify the attack vector and confirm it has been closed
  • Document all findings in an incident log

Phase 5 – Recovery

  • Issue new credentials and verify all access controls are functioning correctly
  • Redeploy affected Cloud Run services from a clean source build
  • Run a full Trivy container and codebase scan to confirm no residual compromise
  • Run Prowler GCP benchmark to confirm infrastructure security posture
  • Monitor GCP Cloud Audit Logs closely for 7 days post-incident

Phase 6 – Post-Incident Review

Within 14 days of resolving a P1 or P2 incident, a post-incident review is documented covering root cause, timeline, effectiveness of response, and actions to prevent recurrence.
